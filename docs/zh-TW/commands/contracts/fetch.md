# contracts fetch

## 功能定位

這是一個底層維護命令：**強制下載更新商品檔 (Fetch Contracts)**。
Shioaji 的所有操作 (特別是下單與報價訂閱) 都極度依賴本機端的「商品檔快取」。商品檔定義了今天所有的股票、期權合約的合法代碼、漲跌停價與結算日。此命令會強迫 API 不管快取狀態，直接向券商伺服器重新拉取最新的一天份商品資料。

## 適用情境

- 剛過午夜 00:00，準備接收下一個交易日的最新商品參數（例如更新隔天的除權息平盤價、或新的選擇權週選合約）。
- 當你執行 `orders place` 或 `quote subscribe` 時，系統報錯提示「`Contract not found`」(找不到合約)，但你很確定代碼打對了。
- 在全域參數中設定了 `--contracts-auto-download false` 阻止啟動時自動下載商品檔，而在需要時才透過此指令手動下載。

## 命令語法

```powershell
python scripts/shioaji_cli.py contracts fetch
```

## 參數詳解

| 參數 | 必填 | 預設值 | 說明 |
|---|---|---|---|
| `--contract-download` | 否 | `true` | 在底層是否確實觸發非同步下載。設為 `false` 在此命令下無太大意義，預設即可。 |

---

## 執行邏輯與背後機制

此命令底層呼叫的是 `api.fetch_contracts()`。
這是一個**非同步**的網路 I/O 操作。當你呼叫後：
1. 主程式會對伺服器發射一個龐大的資料請求。
2. 伺服器會分批把數十萬筆台股與期權的 Contract 資料倒進你的 `api.Contracts` 記憶體中。
3. 命令回傳時，資料可能**還在背景繼續寫入**。所以你不能假設這個指令結束的下一毫秒，商品檔就已經 100% 準備好。

## 實戰與進階範例

### 1. 手動更新商品檔並接續查詢確認
如果你是寫批次檔，正確的防呆做法是更新後，再去查一下 `status` 確認它真的做完了：

```powershell
# 第一步：觸發下載
python scripts/shioaji_cli.py contracts fetch

# 第二步：檢查狀態 (建議寫個 while loop 檢查，直到 output 顯示已下載完畢)
python scripts/shioaji_cli.py contracts status
```

## 注意事項與雷區

1. **網路頻寬消耗**: `fetch` 動作其實非常消耗網路頻寬與券商伺服器資源，因為資料量龐大。**一天只要執行一次就夠了**，沒有特殊狀況請勿在盤中反覆 `fetch`。
2. **自動下載機制**: 其實，只要你在全域參數中**沒有**帶入 `--contracts-auto-download false` (預設為 true)，每次 CLI 啟動登入成功那一瞬間，背景就已經在幫你下載商品檔了。這也是為什麼很多時候你不需要手動呼叫此命令的原因。
3. **換日的時間點**: 台灣時間的凌晨過後，交易所才會開始切換系統並結算隔日資料。如果你在 00:01 太早 fetch，是有可能還抓到舊的、昨天的合約資料。通常實務上會在早上 06:00 到 08:00 間（例如每日自動喚醒模組）才執行商品檔更新。

1. 先確認商品檔狀態。
2. 必要時強制下載商品檔。
3. 再查詢代碼或進行下單。
